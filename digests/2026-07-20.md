## 1. MCP仕様 2026-07-28リリース候補 — ステートレス化・MCP Apps・Tasks拡張

**情報源:** [Model Context Protocol Blog](https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/)（公開日: 2026年5月21日、著者: David Soria Parra / Den Delimarsky, Lead Maintainers）／最終仕様公開予定は2026年7月28日

**何が新しいか:**
MCP発足以来「最大級」と位置づけられる改訂。プロトコルコアからセッション概念とinitializeハンドシェイクを取り除き、すべてのリクエストを自己完結させる「ステートレス」設計に変更する。あわせてサーバー側レンダリングUIを可能にする「MCP Apps」、長時間実行タスクを扱う「Tasks」拡張が正式な仕様として導入される。

**技術的なポイント:**
- ステートレス化により、これまでスティッキーセッションや共有セッションストアが必要だったリモートMCPサーバーが、単純なラウンドロビン型ロードバランサ配下でも動作可能に。`tools/list`のレスポンスをクライアント側でキャッシュできるようになる。
- MCP Appsはツールがあらかじめ「UIテンプレート」を宣言し、ホスト側がサンドボックス化されたiframeで事前フェッチ・キャッシュ・セキュリティレビューできる仕組み。
- Tasks拡張は`tools/call`がタスクハンドルを返し、クライアントが`tasks/get`/`tasks/update`/`tasks/cancel`で制御する非同期モデル。
- 認可まわりはOAuth 2.0/OpenID Connectとの整合性を強化し、「Enterprise-Managed Authorization」拡張が安定版に昇格（IdP経由での一元アクセス制御）。
- Roots・Sampling・Loggingは非推奨化。機能ライフサイクルポリシーにより、非推奨から削除まで最低12ヶ月の猶予が保証される。
- JSON Schema 2020-12への対応でツール定義がより柔軟に。

**なぜ重要か:**
MCPサーバーを自作・運用する開発者にとっては破壊的変更を含むため、SDK側は10週間の検証期間を設けている。FDE的にAIエージェントを企業に導入する立場では、認可の標準化（OAuth/OIDC整合）とEnterprise-Managed Authorizationは特に重要で、企業向けMCPサーバー導入時の「誰にどのツールを許可するか」という運用課題に直結する。ステートレス化はスケーラビリティ面でも実務上のインパクトが大きい。

**リンク:** https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/

## 2. UniClawBench: A Universal Benchmark for Proactive Agents on Real-World Tasks

**情報源:** arXiv (2026年7月9日提出), 著者: Zhekai Chen, Chengqi Duan, Kaiyue Sun, Bohao Li, Yuqing Wang, Manyuan Zhang, Xihui Liu

**何が新しいか:**
「proactiveエージェント」（ユーザーの日常ツールを操作し実世界で支援するエージェント）を評価するための、能力分解型ベンチマーク。既存ベンチマークがサンドボックス環境・単発ターン評価に偏り、失敗原因を切り分けにくい問題を指摘し、5つの基礎能力軸でタスクを再設計している。

**技術的なポイント:**
- 5つの基礎能力: Skill Usage（ツール利用）、Exploration（探索）、Long-Context Reasoning、Multimodal Understanding、Cross-Platform Coordination。これらに基づき400件のバイリンガル実世界タスクを構築。
- 静的な正解データに頼らず、Dockerコンテナ上のライブ環境でエージェントを動かし、細粒度のステップ単位チェックポイントで評価。
- 3ロールのクローズドループ評価フレームワーク: 実際にタスクを実行する「executorエージェント」、隠しルーブリックで軌跡・成果物を評価する「supervisor」、自然なフィードバックを返す「user simulator」。

**なぜ重要か:**
単発のQ&Aベンチマークではなく、複数ツール・複数プラットフォームをまたいで自律的に動くエージェントの「どこで失敗するか」を切り分けられる設計は、実務でエージェントを構築・デバッグする際の評価設計の参考になる。FDE的にクライアント環境でエージェントを組む際、こうした能力分解の考え方（探索能力とツール利用能力を別軸で見る）はそのまま自社の評価基盤設計に応用できる。

**リンク:** https://arxiv.org/abs/2607.08768

## 3. OpenWiki（LangChain） — コーディングエージェント向けにリポジトリ知識を自動維持するCLI

**情報源:** GitHub — [langchain-ai/openwiki](https://github.com/langchain-ai/openwiki)（LangChain公式、MITライセンス）

**何が新しいか:**
コードベース用のドキュメント（wiki）をエージェント駆動で自動生成・自動更新し続けるOSS CLI。コーディングエージェントが1つの巨大なinstructionファイルを毎回全部読み込まなくても、必要なリポジトリ文脈を参照できるようにする狙い。

**技術的なポイント:**
- 「code mode」（リポジトリドキュメント生成）と「personal mode」（個人のローカル知識wiki構築）の2モード。
- Git・Notion・Gmail・X/Twitter・Web検索・Hacker Newsなど複数のコネクタから知識を取り込み可能。
- Google Open Knowledge Format (OKF) v0.1互換で出力。
- GitHub Actions/GitLab CI/Bitbucket Pipelinesに対応し、スケジュール実行で差分更新（前回実行以降のコミットとgit diffを見て更新箇所を判断）。
- モデルプロバイダーはOpenAI・OpenRouter・Gemini・AWS Bedrock・Anthropic・Nebius・Fireworks・Baseten・NVIDIA NIM、およびOllama/LM StudioなどOpenAI互換エンドポイントに対応。
- `npm install -g openwiki` でグローバルインストール、`openwiki --init`で初期化。

**なぜ重要か:**
コーディングエージェント（Claude Code、Codex CLI等）の性能はリポジトリ文脈の質に強く依存する。CLAUDE.md的な単一ファイルでは大規模リポジトリの文脈を維持しづらいという課題に対する具体的な解の一つで、社内の複数リポジトリにAIエージェントを導入するFDE的な仕事では、こうした「エージェント用ドキュメントの自動保守」の仕組みをそのまま提案・導入できる。

**リンク:** https://github.com/langchain-ai/openwiki
